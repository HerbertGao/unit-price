# 小程序产品形态(browse-first 比价)

> 本文是 `apps/miniapp` **产品形态与信息架构的单一事实源**。架构总图见 [`architecture.md`](architecture.md);品类树 / 标签 / 对比组机制见 [`taxonomy-and-tagging.md`](taxonomy-and-tagging.md)(本文**消费**其模型,不复制)。本文**细化并部分取代** `architecture.md` §六.2 对小程序的描述(由「榜单浏览 + 手动/扫码录入」收敛为「浏览优先 3-Tab,录入降级为搜索兜底」)。后续对小程序的任何改动以本文为参考。

## 一、定位与核心判断

`apps/miniapp` 是一个**浏览优先(browse-first)的中国大陆比价小程序**:用户**先看我们的榜单 / 品类树,再决定买哪个**(行前规划场景),而**不是**在货架前「拿着商品即时查单价」的 on-demand 决策伴侣。

形态由一条约束链推导而来,理解它才能避免日后改动时走偏:

```
不能服务端爬山姆(合规红线)
  → 数据「先抓好再展示」(运营预先入库,前端读目录)
  → 产品可信度 = 目录完整度 × 价格新鲜度
  → 供数侧(运营定期自抓 HAR 批量 ingest)是产品的存亡问题,非副作用
```

因此 **on-demand `/parse` 不是主角**——它降级为「搜索未命中兜底 + 贡献入口」(见 §六)。

> v1 垂直域(软饮)有一个关键事实:**山姆软饮品类本就少,现有数据已是全量**。故上述链条中「完整度 × 新鲜度」对 v1 的活跃约束主要落在**新鲜度**;供数的持续职责是刷新价格、而非补广度(详见 §四)。v1 只针对软饮做横向对比,不强求件数。

护城河仍与 `architecture.md` 一致:**AI 把脏规格结构化 + 确定性程序算可比单价 + 可比性判断 + 可回放 `formula`**。差异化不来自「爬到的价多」或「目录大」,而来自**每条结论都诚实、可解释、敢说「这件不可比 / 暂不排名 / 数据可能已变动」**。

## 二、信息架构(3 个底部 Tab)

贴国内比价 / 电商小程序惯例,采用 **榜单 / 分类 / 我的** 三 Tab 主干。**搜索不单独占 Tab**,以**顶部常驻搜索栏**形式存在于榜单与分类页(国内惯例)。

| Tab | 定位 | 关键元素 |
|---|---|---|
| **榜单(首页)** | 发现入口、反套路 payoff | 顶部常驻搜索栏;旗舰「软饮真实单价榜 · 元/100ml」;attribute chips(无糖 / 气泡 / 进口,点击即时重排**同一 cohort**);品类入口卡(碳酸 / 果汁植物饮 / 咖啡茶饮 / 饮用水);**新鲜度 + 范围诚实横幅**(「含山姆软饮 N 件 · 数据更新于 X月X日 · 酒类暂不参与排名」) |
| **分类(品类树)** | 按结构浏览,树里天然按单价排序 | 左栏 category is-a 树;右栏选中节点的 cohort,按节点继承的 `comparable_unit` **升序**;顶部 attribute 过滤 chips(动态复合查询);不可排名节点明示「暂不支持横向对比」 |
| **我的** | 账户 + 众包 + 信任叙事 | 贡献入口(粘标题+价,见 §六);收藏的榜单 / 商品;纠错入口(`/corrections`);「我们怎么算单价」方法说明 |

**Tab 取舍**:搜索若单列为 Tab 不符国内惯例、且会挤占主干;贡献在零量阶段不值一个 Tab(会是空房间),降为「我的」内 CTA。3 Tab 即可承载六项能力(榜单 / 品类树 / 标签筛选 / 品名搜索 / 详情同品类比价 / 贡献)。

## 三、对比组与排序(消费 taxonomy 模型)

榜单与详情的「同品类比价」统一建立在 [`taxonomy-and-tagging.md`](taxonomy-and-tagging.md) 的模型上,本文不复制,仅约定前端口径:

- **一个 cohort = 一个 category 节点的 is-a 闭包 [∧ attribute 过滤]**,按该节点解析出的 `comparable_unit` **升序**排名。
- `comparable_unit` 绑在「软饮」、叶子继承;前端展示的「每升 / 每瓶」由 per100ml + 规格**前端换算**派生,排名主单位恒为存储的 per100ml。
- **v1 仅 per100ml 软饮节点可排名**;酒类 / 重量品 / 纸品 `rankable=false`(展示策略见 §九)。
- **禁止 null 单位进入排名**;复合查询(无糖碳酸、所有气泡饮品等)**动态**计算、不物化。

## 四、供数策略(广度已就位,持续职责是新鲜度)

浏览优先意味着「目录不全 / 价格陈旧」直接等于产品不可信,而合规上不能爬。但 v1 垂直域有一个决定性事实:**山姆软饮品类本就少,现有数据已是全量**(在售软饮已全部收录)。因此:

- **广度不是问题、也不强求数量**:v1 只针对软饮做横向对比,**不为提升件数补抓**——山姆软饮本就少,「每叶凑够 N 件」无意义。供数的存量目标已达成。
- **持续职责 = 价格新鲜度 + 捕获新上架**:运营定期 HAR / 抓包重抽(对应 `architecture.md` §七「App 抓包 = 个人使用」合规层)的目的从「补全目录」转为「**刷新价格、纳入新上架软饮**」。因 tier1 召回偏低(约 20.7%),入库配 **tier2 AI 解析**提升可算率。
- **众包 = 新鲜度补充**,不指望它建库;最高价值是对已收录商品的**价格新鲜度上报**,按 trickle 设计。
- **上线门槛**(广度已满,故收敛为呈现 + 新鲜度 + 分发):全量软饮呈现到位、价均在新鲜阈内、诚实横幅就位、ICP 域名可达。**不设「每叶 ≥N 件」数量门槛**。低于此只在 dev-API 内部验证,不对外称「上线」。

## 五、价格新鲜度

入库管道为 first-write-wins、**重抽不覆写已落价**(现状见相关持久化约定)。本期**不先建价格刷新**,改用诚实披露兜底:

- 每条价格带**采集时间戳**(来自 `product_raw` 落库 / 观察时间;若现有 schema 无可用字段,这是一处小的 ingest / schema 增项)。
- 超阈(初版 **30 天**,可调)的项打「**可能已变动**」标 + **排序降权**。
- **绝不**把陈旧价当当前价展示。
- 新鲜度主要由「深度抽取本身 = 一次新鲜捕获」保证;**价格刷新 upsert 为未来增项**。

## 六、on-demand `/parse` 的归宿

`/parse` 引擎**不删、降级**,承担浏览优先形态下两个职责:

1. **搜索未命中兜底**(主):搜不到的商品 → 「没找到?粘上山姆标题 + 价,帮你算单价」→ core tier1 即时算 per100ml + 展示 `formula` →「要收录吗」。把目录的**不完整**从死胡同变成优雅能力。
2. **贡献入口**(我的 Tab):同一粘贴流,带「收录到榜单」→ `/contribute`。
3. **需求信号**:未命中的搜索词记录下来,供运营**下一轮抓取的优先级清单**——把端的缺口直接接回供数侧。

它**不是 hero、不占 Tab、不是主流程**;只是让「已知不完整的目录」显得有意为之而非残缺。

## 七、详情页与低样本诚实

详情页交付「同品类比价结果」,且必须在 cohort 仅几件时仍可信:

- 顶部:商品本身——**商品图**(复用列表缩略图、单图,不取额外详情图;见 §十三,可空 / 可关)、名称、规格、**大字 per100ml**、货架价(参考标价、小字)、**采集于 X月X日**、本品在 cohort 中的名次。
- **可回放 `formula` 直接展示**(如 `¥40 ÷ (330ml×24) × 100 = ¥0.51/100ml`)——护城河可视化,不藏在二级页。
- **同品类 cohort 排名**,本品高亮;attribute chips 现场收窄 peer 集(闭包 ∧ attribute)。
- **低样本诚实**:明说「当前共收录 N 件,样本较少,排名仅供参考」,绝不让 4 件的榜显得权威。
- **warnings 透出**(如「数量按单件推断为 1」)作为提示;**不展示 `confidence`**(生产恒约 0.95、无区分力,是伪装成信号的噪声)。
- **`comparable=false`**:展示 `excludedReason`(组合礼包 / 规格缺失 / 赠品复杂 / 促销复杂),并将其**排除出 cohort 排序**,专业地说「这件不参与比价」。

## 八、搜索行为(命中 / 未命中)

品名搜索 + 标签 / 品类过滤,跑在一个**明知不完整**的目录上,核心是**优雅降级**:

- **命中**:解析为复合查询(品名 / 品牌 / 品类 / 属性)→ cohort 行展示(per100ml 排序)。
- **部分命中**:明示「找到山姆 N 件,其他门店暂未收录」,不假装这几件就是全部。
- **未命中**:**绝不**空结果——① 推荐最近的 cohort(「没找到这件,看看碳酸饮料榜?」);② on-demand 兜底(§六);③ 记为需求信号(§六.3)。

## 九、不可排名项(酒类等)的展示

- 酒类等 `rankable=false` 节点**显示但标「暂不支持横向对比」**——可点进看列表,无单价排序。
- 兑现「不追求万物可比」原则:把「不可比 / 暂不排名」当成**专业的答案**,而非缺陷或空白。
- 此策略对齐 [`taxonomy-and-tagging.md`](taxonomy-and-tagging.md) 中酒类子树「展示但 `rankable=false`」的设计。

## 十、排期与分发长杆

- **分发长杆 = 国内可达 + ICP 备案域名**(`workers.dev` 大陆不可达);属日历时间,**day-1 并行启动备案**。
- **域名前**(全部可在微信 devtools + dev-API 内验证,不卡域名):3 Tab 端 UI、品类树、搜索 + 未命中兜底、详情页(formula / cohort / 低样本诚实 / warnings)、价格新鲜度展示。
- **域名后**(ICP 落地后):真机预览 → 上架 → 开真实流量 → 众包 trickle 启动。

## 十一、落地依赖(形态 → build)

本形态**依赖 taxonomy / tagging 基建**(tag 字典含 kind + is-a + `comparable_unit`、`product_tag`、`store_category_map`、`category_closure`、打标签管线),该基建在 [`taxonomy-and-tagging.md`](taxonomy-and-tagging.md) 是 v1 设计但**尚未实现**;现有 `GET /rankings` 只是**单条扁平 per100ml 榜、无品类树**。

落地按依赖顺序拆为多个 OpenSpec 变更:

1. **基建**:Taxonomy v1(tag 字典 + is-a + `comparable_unit`、`product_tag`、`store_category_map`、`category_closure`、tier1 规则 + 山姆原生分类映射的打标签管线)。
2. **API**:品类树 + 节点 cohort 榜(把 `/rankings` 泛化为「按 category 闭包 ∧ attribute 取榜、按节点继承单位排序」);品名搜索(命中 / 部分 / 未命中);详情 cohort。
3. **端**:3 Tab Taro(榜单首页 / 分类树 / 详情页 / 我的),消费 `@unit-price/api-client` 预构建 dist。
4. **供数**(运营动作,非代码):软饮已全量,无需为广度补抓;定期 HAR 重抽**刷新价格 + 纳入新上架软饮**。
5. **商品图管线**:见 §十三(从 COS 拉 `imageMogr2` 缩略 → R2,经 ICP 域名提供;`image` 全程可空 / 可关 / 非承载)。

**v1 明确不做(proudly say no)**:多商店、酒类排名、per100g / 纸品计算、LLM 候选打标签、价格刷新 upsert、扫码 / 拍照 / OCR 录入。

## 十二、核心原则(对齐 `architecture.md` §十,落到小程序)

1. 不追求万物可比——`comparable=false` / `rankable=false` 明确标注,是专业感。
2. 默认最稳单位——软饮 per100ml;展示派生单位由 per100ml 换算。
3. AI 只理解、不计算——单价、换算、可比与否由规则引擎与确定性程序决定。
4. 每个结论可解释——详情页直接展示可回放 `formula`。
5. 先窄——v1 = 山姆 + 软饮 + per100ml,深耕单店,跨店为 v2。

## 十三、商品图与版权(知情承担风险)

**决定**:v1 **自托管**山姆列表商品图,换取比价小程序应有的辨识与观感。**已知此为产品中 IP 风险最高的资产**:自托管使我方成为信息网络传播权意义上的传播者,与「不爬、不搬运」的合规姿态相抵,微信侵权投诉可快速下架。本节护栏是「承担风险」的前提,后续改动不得削弱。

**范围(最小)**:每个商品**只存一张列表缩略图(单图)**;**不做商品详情图 / 图集 / 氛围图**。详情页复用这张列表缩略图。

**数据源(已由 HAR 验证)**:列表接口 `api/v1/sams/goods-portal/grouping/list` 每个商品项带 `image` / `thumbnailImage`(腾讯云 COS,`sam-material-online-*.file.myqcloud.com/sams-static/goods/{goodsId}/…jpg`);COS 支持 `?imageMogr2/thumbnail/!…` 在线缩放,故**取图时直接拉已缩小的缩略图**,无需自建缩放。忽略 `images[]` 多变体、`atmosphereImages`、`logoImage*`(配送角标)。

**托管 = Cloudflare R2**(贴现有 CF 栈、低成本):
- 已全在 Cloudflare(Workers + D1);R2 原生 Workers 绑定(`env.BUCKET`)、**零出口流量费**、免费额度足以覆盖小全量软饮(数百张 × 数十 KB = 数 MB)。
- 管线:ingest 侧从 `image` 拉 `?imageMogr2/thumbnail/!…` 缩略 → 以稳定 key(`goods/{goodsId}.jpg`)`put` 进 R2 → 经 Worker 路由 / R2 自定义域**从 ICP 域名**提供(CDN 缓存)。下采样交 COS,**不需 Cloudflare Images**(付费)。
- **不引入新阻塞**:`r2.dev` 与 COS 源站在大陆同样不可靠,图必须从 **ICP 域名**出——与 API 同一个长杆,不额外加一个。

**护栏(必须随图实现,降低下架爆炸半径)**:
1. **服务端总开关 + 图非承载**:`image` 在端 / API / schema 全程**可空、可关、非承载**;UI 无图时优雅降级回文字优先(品类图标 + 品牌名 / 色块,§二 / §七),不破版。收投诉 / 律师函 → 翻开关,**分钟级回退、免小程序重审**。
2. **通知-删除流程 + provenance**:留版权联系入口 + 快速移除;每图存来源 URL / 商店 / 采集时间,支持**按品牌 / 按图单点下架**。
3. **只存缩略图**(非全幅原图);作商品识别图、非品牌横幅。
4. **限软饮**(v1 范围),暴露面有限。

**build 蕴含**:ingest 侧加图片步骤(拉缩略 → put R2);`product` / 响应 schema 加可空 `imageKey`;端按 §二 / §七 渲染,缺图走降级。
